# คู่มือการพัฒนาโปรเจกต์ "เมียงขนาด" (Development & Knowledge Guide)

เอกสารนี้รวบรวมขั้นตอน การทำงาน และ **ความรู้พื้นฐาน (Knowledge)** ที่สำคัญในการสร้างโปรเจกต์นี้ เพื่อเป็นคัมภีร์สำหรับการพัฒนาต่อในอนาคต

---

## 🏗️ Phase 1: สถาปัตยกรรมระบบ (System Architecture)

### 1. การแบ่งส่วนงาน (Frontend vs Backend)
- **Frontend (Client)**: คือส่วนที่ผู้ใช้เห็น (React). หน้าที่คือการแสดงผล UI และส่งคำขอ (Request) ไปยัง Server
- **Backend (Server)**: คือส่วนประมวลผล (Node.js). หน้าที่คือการคุยกับฐานข้อมูล และตอบกลับ (Response) ข้อมูลที่ปลอดภัยให้กับหน้าบ้าน
- **ความรู้ (Knowledge)**: ทั้งสองส่วนแยกกันทำงานอิสระ เชื่อมต่อกันด้วย **API URL**

### 2. ตัวแปรสภาพแวดล้อม (.env)
- ใช้เก็บข้อมูลความลับ เช่น รหัสผ่าน หรือ URL ของฐานข้อมูล
- **กฎสำคัญ**: ห้าม Commit ไฟล์ `.env` ขึ้น Git (ใส่ชื่อใน `.gitignore`) เพื่อความปลอดภัย

## 🌐 Phase 2: เจาะลึกความรู้เรื่อง API (RESTful API)

API คือตัวกลางที่ทำให้หน้าบ้านและหลังบ้านคุยกันได้ โดยมีหลักการดังนี้:

### 1. Method (คำสั่ง)
- **GET**: ใช้สำหรับ "ขอข้อมูล" (เชน ดึงรายการสินค้า)
- **POST**: ใช้สำหรับ "เพิ่มข้อมูลใหม่" (เช่น เพิ่มสินค้า, บันทึก Log)
- **PATCH**: ใช้สำหรับ "แก้ไขข้อมูลบางส่วน" (เช่น เปลี่ยนราคาสินค้า)
- **DELETE**: ใช้สำหรับ "ลบข้อมูล"

### 2. Status Codes (รหัสสถานะ)
คือสิ่งที่ Server บอกกลับมาว่าคำขอสำเร็จไหม:
- `200 OK`: สำเร็จ
- `201 Created`: สร้างข้อมูลใหม่สำเร็จ
- `401 Unauthorized`: ไม่ได้รับอนุญาต (รหัสผ่านผิด/ไม่ได้ Login)
- `400 Bad Request`: หน้าบ้านส่งข้อมูลมาผิดรูปแบบ
- `500 Server Error`: มีข้อผิดพลาดที่ตัว Code หรือ Database

### 3. JSON Structure
ข้อมูลที่รับส่งกันจะอยู่ในรูปแบบ **JSON** เสมอ เพราะใช้ง่ายและเป็นมาตรฐาน:
```json
{
  "success": true,
  "data": { "name": "พิณ", "price": 1500 }
}
```

## 🗄️ Phase 3: การจัดการข้อมูล (Database Knowledge)

### 1. ความสัมพันธ์ของตาราง (Relational Database)
- **Foreign Key**: การใช้ ID เชื่อมโยงกัน เช่น ตาราง `Items` เก็บ `item_type_id` เพื่อไปดึงชื่อหมวดหมู่จากตาราง `item_type`
- **Why?**: เพื่อลดความซ้ำซ้อนของข้อมูล (ถ้าหมวดหมู่เปลี่ยนชื่อ เราแก้ที่เดียว ไม่ต้องตามแก้ในสินค้าทุกชิ้น)

### 2. ตัวระบุตัวตน (ID & Timestamps)
- **Primary Key (ID)**: สินค้าทุกชิ้นต้องมีเลข ID ที่ไม่ซ้ำกันเด็ดขาด
- **created_at**: บันทึกเวลาที่ข้อมูลถูกสร้างขึ้นอัตโนมัติ ช่วยในการเรียงลำดับสินค้าใหม่-เก่า

## 🛡️ Phase 4: ความปลอดภัย (Security Best Practices)

ความปลอดภัยคือเรื่องที่สำคัญที่สุดในการทำ API:

### 1. Validation (การตรวจสอบข้อมูล)
- **กฎ**: อย่าเชื่อใจข้อมูลที่ส่งมาจากหน้าบ้าน
- **วิธี**: ใช้ `express-validator` เช็คเสมอว่าข้อมูลที่ส่งมาเป็นตัวเลขจริงไหม หรือว่างเปล่าหรือเปล่า ก่อนจะส่งลงฐานข้อมูล

### 2. Server-Side Check
- การเช็ครหัสผ่านต้องทำที่ **Backend เท่านั้น** เพราะโค้ดใน Backend ผู้ใช้ปกติจะมองไม่เห็นและแก้ไขไม่ได้

### 3. SQL Injection
- คือการแอบฝังสคริปต์ SQL มาในช่องกรอกข้อมูล
- **วิธีป้องกัน**: ใช้เครื่องมืออย่าง Supabase Query Builder ซึ่งจะทำการ "เตรียมคำสั่ง" ไว้ก่อน (Prepared Statements) ทำให้สคริปต์ที่แอบส่งมากลายเป็นเพียงข้อความธรรมดาที่ไม่ทำงาน

## 📈 Phase 5: สรุปเทคนิคเฉพาะ (Expert Tips)

1. **Async/Await**: การคุยกับ Database ต้องใช้เวลา เราจึงต้องใช้คำสั่งเหล่านี้เพื่อบอกโปรแกรมว่า "รอข้อมูลกลับมาก่อนนะ" ป้องกันแอปค้าง
2. **Error Handling**: การใส่ `try/catch` คลุมโค้ดที่อาจจะพัง (เช่น เน็ตหลุด) จะช่วยให้เราแสดงข้อความเตือนผู้ใช้ได้อย่างสวยงาม แทนที่จะปล่อยให้หน้าเว็บขาว
3. **Clean Comments**: 
    - Frontend: คอมเมนต์ภาษาไทยเพื่อบอก "หัวใจ" ของธุรกิจ
    - Backend: คอมเมนต์ภาษาอังกฤษเพื่อความ "เป็นมืออาชีพ" และตามมาตรฐานสากล

## 🖼️ Phase 6: การจัดการรูปภาพ (Asset Management Knowledge)

คำถามสำคัญ: **"เราควรเก็บรูปภาพสินค้าไว้ที่ไหน?"**

### 1. เก็บใน `public` folder (ไม่แนะนำสำหรับสินค้าจริง)
- **ข้อดี**: ใช้ง่าย แค่ลากลงโฟลเดอร์แล้วเรียก path ได้เลย
- **ข้อเสีย**: เมื่อมีสินค้าเยอะขึ้น แอปจะบวมและโหลดช้ามาก และ **Admin ไม่สามารถอัปโหลดรูปใหม่ได้** โดยไม่แก้โค้ดและ Deploy เว็บใหม่

### 2. เก็บใน Cloud Storage (เช่น Supabase Storage) - **แนะนำ**
- **วิธีทำงาน**: เมื่อ Admin เลือกรูป ระบบจะอัปโหลดไปที่ Storage และเก็บ **URL** ของรูปนั้นลงในฐานข้อมูล
- **ข้อดี**: 
    - **Scalability**: เก็บรูปได้เป็นหมื่นรูปโดยไม่ทำให้เว็บช้าลง
    - **Dynamic**: Admin สามารถเพิ่ม/ลบรูปได้เองทันทีผ่านหน้า Dashboard
    - **Performance**: รูปจะถูกโหลดผ่าน CDN ซึ่งเร็วกว่าการโหลดจากตัวแอปโดยตรง

### 3. ขั้นตอนการอัปโหลด (How the Upload Flow Works)
แม้ว่าคุณจะเป็นคน **"อัปโหลดเอง"** ผ่านหน้า Admin แต่เบื้องหลังระบบจะทำงานแบบนี้ครับ:
1. **หน้าบ้าน (Browser)**: คุณเลือกไฟล์จากเครื่อง และกด "บันทึก"
2. **การส่งข้อมูล**: เบราว์เซอร์จะส่งไฟล์นั้นไปที่ **Cloud Storage** (ไม่ใช่ส่งไปเก็บในโฟลเดอร์ของแอป)
3. **การเก็บรหัส**: เมื่ออัปโหลดเสร็จ Cloud Storage จะส่ง **URL** ของรูปกลับมาให้เรา
4. **ฐานข้อมูล (DB)**: เราจะเก็บแค่ **URL (ที่อยู่รูป)** ลงในตารางสินค้า
5. **การแสดงผล**: เมื่อเปิดหน้าเว็บ เบราว์เซอร์จะไป "ดึงรูป" จากที่อยู่นั้นมาแสดงผลทันที

**ทำไมถึงต้องทำแบบนี้?**: เพราะถ้าคุณอัปโหลดไฟล์เข้าไปในโฟลเดอร์ของแอปโดยตรง เมื่อมีการอัปเดตระบบหรือย้าย Server ไฟล์รูปภาพเหล่านั้นอาจจะ **หายไป** ได้ครับ การใช้ Cloud Storage จึงปลอดภัยและยั่งยืนที่สุดครับ

### 4. รายชื่อ Cloud Storage สายฟรี (Free Cloud Storage Options)
หากคุณกำลังมองหาที่ฝากรูปฟรี มีตัวเลือกที่น่าสนใจดังนี้:
- **Supabase Storage (แนะนำ)**: ฟรี 1 GB, อยู่ในโปรเจกต์เดียวกับ DB ทำให้จัดการง่ายที่สุด
- **Cloudinary**: เหมาะสำหรับสายแต่งรูป เพราะปรับขนาด/ฟิลเตอร์รูปได้ผ่าน URL โดยตรง
- **Uploadcare**: มีระบบอัปโหลดที่สวยงามและใช้งานง่ายมากสำหรับมือใหม่
- **Firebase Storage**: ให้พื้นที่เยอะ (5 GB) แต่การตั้งค่าอาจจะซับซ้อนกว่าตัวอื่น

## 🎨 Phase 7: การออกแบบและการใช้ไอคอน (UI & Iconography Knowledge)

ในการทำเว็บระดับมืออาชีพ การเลือกใช้ไอคอนมีผลต่อความรู้สึกของผู้ใช้อย่างมากครับ:

### 1. ระบบ Icon แบบ CSS (LineIcons) vs. รูปภาพ (Image)
- **LineIcons (ที่ใช้อยู่)**: ทำงานเหมือนตัวหนังสือ (Font) ทำให้ขยายขนาดได้โดยไม่แตก และเปลี่ยนสีได้ทันทีผ่าน CSS
- **ข้อดี**: โหลดครั้งเดียวใช้ได้ทุกหน้าเว็บ ประหยัดทรัพยากร และดูเป็นมืออาชีพกว่าการใช้ Emoji

### 2. การสร้าง Wrapper Component (เทคนิค `LIcon`)
- **ทำไปทำไม?**: แทนที่จะเขียน `<span class="lni lni-package"></span>` ซ้ำๆ ทั่วทั้งเว็บ เราสร้างคอมโพเนนต์ `<LIcon />` ขึ้นมาหุ้มไว้
- **ประโยชน์**: 
    - **Global Change**: ถ้าวันหนึ่งอยากเปลี่ยนรุ่นไอคอน เราแก้ที่ไฟล์ `AdminUI.tsx` ที่เดียว โดยไม่ต้องไล่แก้ร้อยจุด
    - **Size Standard**: การตั้งค่าขนาดเป็น `xs`, `sm`, `lg` ช่วยให้หน้าเว็บมีมาตรฐานขนาดที่เท่ากัน (Consistency) ไม่มั่ว

### 3. แนวทางการเลือกไอคอน (Professional Choice)
- เลือกไอคอนที่ **"สื่อความหมายสากล"** เช่น `package` สำหรับสินค้า, `layers` สำหรับหมวดหมู่, `timer` สำหรับประวัติ
- **Less is More**: ไม่จำเป็นต้องมีไอคอนทุกจุด แต่ไอคอนที่มีต้องช่วยให้ผู้ใช้ "กดถูก" ได้เร็วขึ้นโดยไม่ต้องอ่านตัวหนังสือครับ
